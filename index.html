What is a Failover Cluster?
ℹ Simple Definition

A Windows Server Failover Cluster (WSFC) is a group of independent servers (called nodes) that work together so that if one server fails, another automatically takes over — with little or no downtime for users.

Think of it like a relay race. If one runner falls, the next picks up the baton and keeps going. The race (your service) continues even though one runner (server) failed.

The Core Problem It Solves
✗ Without Cluster

• Single server hosts SQL
• Server fails at 2 AM
• SQL is offline
• DBA gets paged, hours of downtime

✓ With Failover Cluster

• 2 nodes both run ClusSvc
• Node 1 fails at 2 AM
• Node 2 takes over in ~5–30s
• Users reconnect, no one paged

Key Terminology
Node NOUN

Any server that is a member of the cluster. Min 2, max 64 nodes. Each runs ClusSvc.

ClusSvc SERVICE

The NT Service (clussvc.exe) — the brain of the cluster. Manages node membership, heartbeats, and failovers. Runs on every node.

Role / Resource Group NOUN

A logical unit of work (SQL Server, VM, File Server). Contains Resources that all move together during failover.

Quorum MECHANISM

The voting system that determines if the cluster is alive. Requires a majority of votes. Prevents split-brain data corruption.

Heartbeat MECHANISM

A 1-second UDP pulse on port 3343 via NetFT.sys. 5 missed heartbeats = node declared dead (~5 seconds).

Witness COMPONENT

A tie-breaking vote: shared disk, file share (\\server\share), or Azure Blob. Essential for 2-node clusters.

CNO — Cluster Name Object AD OBJECT

The AD computer account (CLUSTER01$) created by New-Cluster. The cluster's identity in Active Directory.

VCO — Virtual Computer Object AD OBJECT

A separate AD account for each clustered role (SQL01$). Clients connect to the VCO name — failover is transparent.

CLUSDB DATABASE

The cluster config database — a binary registry hive at %SystemRoot%\Cluster\CLUSDB. Replicated to every node automatically.

Virtual IP (VIP) NETWORK

The IP clients use to connect to clustered services. Migrates between nodes during failover. Lives on a virtual NDIS adapter.

CSV — Cluster Shared Volume STORAGE

Shared storage (NTFS/ReFS) accessible from ALL nodes simultaneously. Mounted at C:\ClusterStorage\VolumeX on every node.

Why Use Failover Clusters?
🛡 High Availability

99.99% uptime SLAs. Automatic recovery without human intervention.

🗄 SQL Server AlwaysOn FCI

Failover Clustering is mandatory for SQL Server Failover Cluster Instances.

💻 Hyper-V HA / Live Migration

VMs can Live Migrate (zero downtime) or fail over between cluster nodes.

🔧 Planned Maintenance Without Downtime

Drain a node, patch it, reboot it, bring it back — cluster-aware updating.

🌐 Disaster Recovery (Multi-Site)

Stretch clusters across data centers for site-level failover.

Cluster Architecture & Internal Components
► ClusSvc — The Cluster Service (clussvc.exe)▾

What it does:

  • Manages node membership — knows which nodes are alive
  • Runs the Failover Manager (FM) — decides where resources live
  • Exposes the Cluster API (clusapi2.dll) — how PowerShell/GUI talk to the cluster
  • Listens on TCP port 135 (RPC endpoint mapper) + dynamic high ports
# Check ClusSvc status
Get-Service ClusSvc | Select Name,Status,StartType
Start-Service ClusSvc
Stop-Service ClusSvc
► NetFT.sys — Network Fault Tolerant Driver▾

A kernel-mode NDIS 6.x Lightweight Filter driver. This is how heartbeats actually travel.

⚡ Firewall Cannot Block Heartbeats

NetFT.sys operates BELOW Windows Firewall (WFP). You literally cannot block cluster heartbeats with firewall rules — by design.

  • UDP port 3343 for heartbeats
  • Each datagram: node ID, sequence number, cluster GUID, timestamp
  • Default: 1 heartbeat/second, 5 missed = node dead (5000ms DeadTimeout)
(Get-Cluster).SameSubnetDelay      # ms between heartbeats (default: 1000)
(Get-Cluster).SameSubnetThreshold  # missed before node removed (default: 5)
► RHS — Resource Hosting Subsystem (rhs.exe)▾

Hosts cluster resource DLLs in a separate process from ClusSvc. If a resource DLL crashes, only RHS restarts — ClusSvc stays running. Fault isolation.

  • clusres.dll — built-in resources
  • sqlres.dll — SQL Server FCI
  • hadrres.dll — AlwaysOn AG
  • LooksAlive() called every 5s (quick check)
  • IsAlive() called every 60s (deep check)
► CLUSDB — Cluster Configuration Database▾
⚡ Critical Fact

CLUSDB is NOT a SQL database. It's a Windows Registry hive loaded into HKLM\Cluster. Uses ESE transaction logging — same as Active Directory's ntds.dit. Abrupt power loss can corrupt it.

  • Replicated to ALL nodes — two-phase commit protocol
  • Any write committed only after majority of nodes acknowledge
  • Loss of CLUSDB on ALL nodes = rebuild cluster from scratch
reg save HKLM\Cluster C:\Backup\CLUSDB.bak /y
Quorum — The Critical Voting System
⚠ What Happens Without Quorum

Cluster SHUTS DOWN immediately. Prevents split-brain — two halves writing to same storage = data corruption.

Quorum TypeWhen to UseVotesSurvives
Node MajorityOdd nodes (3,5,7...)Each node = 1Floor(N/2) failures
Node + Disk WitnessEven nodes + shared storageNodes + disk = 1 eachHalf nodes + disk online
Node + File Share WitnessEven nodes, no shared diskNodes + FSW = 1 eachHalf nodes + FSW accessible
Cloud WitnessMulti-site, no 3rd siteNodes + Azure = 1 eachHalf nodes + internet
💡 The 2-Node Cluster Rule

2 nodes, no witness: threshold = 2 votes needed. One node fails → 1 vote remaining → QUORUM LOST. Entire cluster offline. ALWAYS configure a witness for 2-node clusters.

Set-ClusterQuorum -FileShareWitness \\DC01\ClusterWitness
Set-ClusterQuorum -NodeMajority
Get-ClusterQuorum | Select *

Why You Cannot Add a Domain Controller as a Cluster Node
⚠ Microsoft's Position

KB281662 explicitly states this is UNSUPPORTED in production. Not "not recommended" — unsupported. Microsoft Support will ask you to fix this before helping.

Reason 1: ClusSvc vs NTDS Service Conflict

ClusSvc has FailureActions that can hard-restart a node. NTDS (AD) has strict ESE checkpoint semantics on ntds.dit. An abrupt ClusSvc-triggered restart can corrupt the AD database — requiring authoritative restore from backup.

Reason 2: Circular Dependency Deadlock

The deadlock: Cluster needs AD → DC is a cluster node → cluster service fails → DC enters recovery → AD goes offline → other nodes can't authenticate to reform cluster → cluster can't recover → AD stays offline → infinite loop.

Reason 3: LSASS/Kerberos KDC Lock Collision

KDC on DC uses LSASS for security context management. ClusSvc recovery tries to manipulate security contexts during failover. Lock collisions can deadlock the security subsystem entirely.

Reason 4: SYSVOL/DFSR Filter Driver Conflict

DFSR's dfsr.sys and WSFC's CSVFS filter driver both operate at the file system mini-filter level. Running both creates filter ordering conflicts → Event 4013 (SYSVOL not synchronized).

Reason 5: Security Boundary Violation
⚠ Tier 0 Asset

DCs are Tier 0 — highest privilege machines. ClusSvc runs on the DC and accepts RPC calls from cluster admin accounts. Any ClusSvc privilege escalation = DC compromise.

Reason 6: DSRM Boot Incompatibility

Booting a DC into DSRM (for AD repair) stops ClusSvc. If that DC hosted the quorum disk/witness, the cluster may dissolve during what should be routine AD maintenance.

💡 What To Do Instead

Keep DCs on separate VMs. Cluster nodes should be plain member servers. Lab minimum: 1 DC VM + 2 cluster node VMs — all separate.

Cluster nodes always use static IPs on all NICs. The cluster VIP can be auto-assigned from DHCP, but static is always preferred.

When You MUST Use -StaticAddress
► Multi-Subnet / Multi-Site Cluster▾

If nodes are in different subnets, provide one IP per subnet.

New-Cluster -Name CLUSTER01 -Node Node1,Node2 -StaticAddress 10.1.0.50,10.2.0.50
► No DHCP in Environment▾

Without DHCP, New-Cluster will fail or assign a 169.x.x.x APIPA address.

► Firewall ACL / PTR Record Required▾

Pre-approve the IP in firewall rules or DNS PTR records, then specify it at creation.

New-Cluster -Name CLUSTER01 -StaticAddress 10.0.1.50
Why DHCP is Dangerous for Cluster IPs
⚠ DHCP Risk Scenarios

Lease expiry: DHCP assigns VIP to another machine → clients hit wrong server.
DHCP down during failover: IP resource fails to come online (Event 1069) → HA cluster is NOT available.
IP conflict: Scope runs out, VIP re-issued to a laptop → ARP chaos.

The Cluster Virtual Adapter
💡 Lesser-Known Fact

The VIP does NOT live on a physical NIC. Windows creates a "Microsoft Failover Cluster Virtual Adapter" — a virtual NIC with its own MAC address. After failover to another node, the virtual adapter is recreated there with the SAME IP and MAC (via gratuitous ARP). Switches update within seconds.

ipconfig /all
# Look for: "Microsoft Failover Cluster Virtual Adapter"
CNO Prestaging — AD Object Lifecycle
ℹ What is Prestaging?

Manually creating the CNO computer account in AD before running New-Cluster. Account must be DISABLED. ClusSvc finds it, enables it, sets SPNs.

Normal Flow vs Prestaged Flow
Normal Flow

1. Run New-Cluster
2. ClusSvc calls LDAP to AD
3. Creates CLUSTER01$ in AD
4. Enables account, sets SPNs
5. Registers DNS A record

⚠ Needs Create Computer Objects permission

Prestaged Flow

1. AD Admin creates CLUSTER01$
2. Places in correct OU
3. Leaves DISABLED
4. Grants CNO Full Control over itself
5. Run New-Cluster
6. ClusSvc finds & enables it

✓ Only needs Write on the prestaged object

Why Prestage? (5 Reasons)

1. Least-Privilege: Cluster admin only needs Write on existing object — not "Create Computer Objects" on the OU.

2. Controlled OU Placement: AD admin places CNO in exactly the right OU → correct GPOs applied.

3. IT Governance: Naming standards and change management approval before object creation.

4. Prevents Event 1045: AD permission errors during cluster creation are eliminated.

5. VCO Control: Same benefits apply to each clustered role's VCO.

CNO Prestaging PowerShell
# Step 1: Create disabled CNO account
New-ADComputer -Name "CLUSTER01" `
  -Path "OU=Clusters,DC=corp,DC=com" `
  -Enabled $false

# Step 2: Grant CNO Full Control over itself
$cno = Get-ADComputer -Identity "CLUSTER01"
$adObject = [ADSI]"LDAP://$($cno.DistinguishedName)"
$acl = $adObject.psbase.ObjectSecurity
$rule = New-Object System.DirectoryServices.ActiveDirectoryAccessRule(
    $cno.SID,
    [System.DirectoryServices.ActiveDirectoryRights]::GenericAll,
    [System.Security.AccessControl.AccessControlType]::Allow)
$acl.AddAccessRule($rule)
$adObject.psbase.CommitChanges()

# Step 3: Create cluster (finds and enables the prestaged CNO)
New-Cluster -Name CLUSTER01 -Node Node1,Node2 `
  -StaticAddress 192.168.1.50 -NoStorage

# Step 4: Verify
Get-ADComputer CLUSTER01 | Select Name,Enabled
setspn -L CLUSTER01
New-Cluster Backend Phases
[01] Node Discovery

RPC to each node's ClusCfgSvr on port 135. Checks OS version, feature installed, not already in a cluster, domain membership.

[02] CLUSDB Initialization

Creates binary registry hive at %SystemRoot%\Cluster\CLUSDB. Writes ClusterName, ClusterGUID, NodeList. Replicates to all nodes via SMB.

[03] CNO Creation/Activation in AD

LdapAdd (new) or LdapModify (prestaged). Enables account, sets SPNs: MSServerCluster/CLUSTER01 and FQDN variant. Generates 120-char random password.

[04] NetFT.sys Binding

Loaded at IRQL PASSIVE_LEVEL. Cluster NICs bound at NDIS 6.x filter layer. Heartbeat UDP socket opened on 3343. Timer set to 1-second interval.

[05] Quorum Resource Creation

Quorum type selected based on node count and storage. Resource group registered in CLUSDB. FM calculates initial vote counts.

[06] IP Resource + DNS Registration

IP Address resource created via clusres.dll COM object. NIC configured with VIP via iphlpapi.dll. DNS A record registered (TTL 1200s).

Cluster Backend Mechanics
Failover Sequence Timeline
ℹ What Happens When a Node Fails

Complete timeline from failure to role back online.

T+0s

Node failure (hardware crash, BSOD, power loss, cable pulled)

T+1s

Other nodes notice: first missed heartbeat on UDP 3343

T+5s

5 consecutive missed heartbeats → node marked Down → Event 1135 logged on surviving nodes

T+5s

Failover Manager triggered — evaluates quorum. Picks best node using PreferredOwners, anti-affinity, available resources.

T+6–20s

IP Address resource Online on new node. Gratuitous ARP sent → switches update MAC tables. VIP live.

T+7–30s

Network Name resource Online. DNS re-registered. SQL Server / application starts.

T+15–60s

Application fully online. Dependent resources started in dependency order.

T+varies

Clients reconnect. SQL MultiSubnetFailover=true reconnects in ~5s. Others wait for TCP timeout (30–120s).

CSV — Cluster Shared Volume
ℹ How CSV Works

Coordinator Node: One node owns CSV metadata. Handles creates/deletes/directory listings. Block I/O goes directly from any node to storage.

Direct I/O Mode: Normal. Fast. Block I/O straight to storage array.

Redirected I/O Mode (Event 5120): All I/O routed through coordinator over SMB. Slower but still running. NOT a failure — a fallback.

Get-ClusterSharedVolume | Select Name, State
(Get-ClusterSharedVolume).SharedVolumeInfo | Where FaultState -ne Normal
Move-ClusterSharedVolume -Name "Cluster Disk 1" -Node Node2

Troubleshooting Common Issues
ℹ Always First

1. Check FailoverClustering/Operational log. 2. Run Get-ClusterLog -UseLocalTime. 3. Check System log on ALL nodes. 4. Test-NetConnection between nodes.

Click any issue to expand the fix:

⚠ Access Denied — Event 1045 (HRESULT 0x80070005)▾

Symptoms: New-Cluster fails immediately. Event 1045 in FailoverClustering/Operational.

Causes:

  • Account lacks "Create Computer Objects" in target OU
  • Prestaged CNO not granted Full Control to itself
  • Kerberos SPN conflict — another object has MSServerCluster/CLUSTER01
  • PowerShell not run as Administrator (UAC)
whoami /priv
dsacls "OU=Clusters,DC=corp,DC=com" | findstr /i "create computer"
setspn -Q MSServerCluster/CLUSTER01
# Fix SPN conflict:
setspn -D MSServerCluster/CLUSTER01 WrongObject
⚠ Node Cannot Join — Event 1254 (HRESULT 0x6D9)▾

Symptoms: Add-ClusterNode fails. "No more endpoints available from endpoint mapper."

Causes:

  • Firewall blocking TCP 135 or UDP/TCP 3343
  • ClusSvc not Running on joining node
  • Stale HKLM\Cluster from previous cluster (not properly evicted)
  • WinRM not enabled
Test-NetConnection -ComputerName NODE2 -Port 3343
Test-NetConnection -ComputerName NODE2 -Port 135
Get-Service ClusSvc -ComputerName NODE2
Enable-PSRemoting -Force
# If stale registry (backup first!):
Remove-ClusterNode -Name NODE2 -Force
# On NODE2: reg delete HKLM\Cluster /f
⚠ Quorum Loss — Cluster Offline — Event 1177▾

Symptoms: Cluster fully offline. All roles down. Event 1177 on remaining nodes.

Causes:

  • 2-node cluster with no witness — one failure = quorum lost
  • FSW share offline (DC rebooting, share deleted, permissions changed)
  • Cloud witness: HTTPS to Azure Blob blocked
# Fix witness
Set-ClusterQuorum -FileShareWitness "\\DC01\ClusterWitness"
Test-Path "\\DC01\ClusterWitness"

# BREAK-GLASS only — only when other nodes confirmed OFF:
Start-ClusterNode -Name NODE1 -FixQuorum
⚠ DNS Registration Failure — Event 1196▾

Symptoms: VIP online but clients can't connect by name. Event 1196.

  • DNS zone "Secure only" but CNO$ lacks DNS write permission
  • Stale DNS record points to old IP
  • DNS server unreachable
Resolve-DnsName CLUSTER01.corp.lab
Remove-DnsServerResourceRecord -ZoneName "corp.lab" -RRType A -Name "CLUSTER01" -Force
# Force re-registration:
Get-ClusterResource | Where ResourceType -eq "Network Name" | Restart-ClusterResource
⚠ CSV Redirected I/O — Event 5120▾

Note: Event 5120 is NOT a failure — it's a degraded fallback mode. I/O continues via coordinator.

  • MPIO path failure — one path to storage failed
  • Antivirus scanning CSV files
  • Disk I/O errors (check Event 11 in System log)
(Get-ClusterSharedVolume).SharedVolumeInfo | Select Name,FaultState
Move-ClusterSharedVolume -Name "Cluster Disk 1" -Node Node2
Get-WinEvent -LogName System | Where {$_.Id -eq 11} | Select -First 5
⚠ Cluster Validation Storage Failure▾
  • iSCSI not connected (MSiSCSI not started)
  • Disk not initialized (RAW partition style)
  • Disk has drive letters assigned
  • MPIO not configured
Get-Service MSiSCSI
Get-Disk | Select Number,OperationalStatus,PartitionStyle,Size
Get-Disk | Where IsOffline | Set-Disk -IsOffline $false
Get-Disk | Where PartitionStyle -eq RAW | Initialize-Disk -PartitionStyle GPT

Common Questions & Doubts — Q&A
ℹ For Presentations

These are the questions that come up most often from technical colleagues.

Q1: What's the difference between a node and a role?▾

A node is a physical/virtual server — it provides compute resources and runs ClusSvc. A role is a workload that runs ON the cluster and can move between nodes. Think: nodes are stages, roles are performers. The performer can move to any available stage if their current one catches fire.

Q2: If I have 3 nodes, do I still need a witness?▾

No. 3 nodes = 3 votes. Quorum needs (3/2)+1 = 2 votes. Lose 1 node → 2 votes remain ≥ 2 → cluster stays online. For 3+ odd nodes, Node Majority (no witness) is fine. For even node counts (2, 4, 6), always add a witness.

Q3: Can the File Share Witness be on a cluster node?▾

Technically yes, but it defeats the purpose entirely. If Node1 hosts the FSW and Node1 fails, you lose Node1's vote AND the witness vote simultaneously → cluster loses quorum. The FSW MUST be on an independent server (DC, management VM, utility server) — NOT a cluster node.

Q4: What is 'Drain Roles' and when should I use it?▾

Suspend-ClusterNode -Drain gracefully migrates all running roles OFF a node BEFORE maintenance. Without draining: reboot = roles failover (brief interruption). With draining: VMs Live Migrate (zero interruption). Always drain before patching, rebooting, NIC driver changes, or firmware updates.

Q5: What's the difference between failover and live migration?▾

Failover = unplanned/reactive. Node dies → cluster detects → roles restart on another node. 5–60 seconds of downtime.

Live Migration = planned/proactive. Node is healthy, you choose to move a VM. Memory copied while VM runs. Cutover <1 second. Zero downtime. Requires the source node to be healthy — can't live-migrate from a dead node.

Q6: What does Force Quorum (FixQuorum) do? When is it safe?▾
⚠ Danger

If used while the other partition is STILL RUNNING → split-brain → both halves write to same storage → guaranteed data corruption.

FixQuorum forces a node to start the cluster ignoring quorum requirements. ONLY safe when you are 100% certain the other nodes/partition are powered OFF — not just unreachable. Use it, verify nothing else is running, then fix the root cause.

Q7: Why doesn't the role come back to its preferred node after recovery?▾

AutoFailbackType = 0 (default = Prevent Failback). The cluster intentionally leaves roles on their current owner after a failover — prevents flip-flopping if the preferred node has intermittent issues.

(Get-ClusterGroup "SQL").AutoFailbackType = 1  # Allow failback
(Get-ClusterGroup "SQL").FailbackWindowStart = 2  # 2 AM window
(Get-ClusterGroup "SQL").FailbackWindowEnd   = 4  # end 4 AM

Q8: What is split-brain and how does quorum prevent it?▾

Split-brain = network partition divides cluster into two groups, both think they're the sole survivor and try to own shared storage. Both write simultaneously → data corruption.

Quorum requires a MAJORITY of votes to stay online. With 2 nodes: each half has 1 vote, neither has majority (need 2), both shut down → no split-brain. The witness provides the tiebreaker, giving one half the majority.

Q9: Can I add/remove nodes from a running cluster without downtime?▾

Yes. Add-ClusterNode adds a new node while the cluster is fully running — existing roles continue uninterrupted. Remove-ClusterNode -Evict removes a node cleanly (drain roles first). Neither operation requires cluster downtime.
